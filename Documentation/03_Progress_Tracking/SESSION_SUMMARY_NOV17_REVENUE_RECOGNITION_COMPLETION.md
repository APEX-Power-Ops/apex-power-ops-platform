## KEY DECISIONS/INSIGHTS

### **Technical Insights**
1. **Calculated Fields Require All Source Data**: Dataverse calculated fields using IF/THEN/ELSE logic return $0 (not null) when source fields are null. This caused "no rates defined" error instead of obvious null reference.

2. **OData Binding Format is Critical**: Power Automate requires specific format for Dataverse lookups: `entitysetname(guid)` not plain GUIDs. Entity set names follow Dataverse pluralization rules (add 'es' for words ending in 's').

3. **Empty() vs Equals("")**: For null checking in Power Automate conditions, `empty()` function works for null/empty, while `equals ""` only checks empty string (not null).

4. **Rollup Helper Fields Cause Import Conflicts**: The 14 helper fields (_date, _state) generated by Dataverse for rollup fields create duplicate column errors on import. These are system-managed and shouldn't be in customizations.xml.

### **Process Insights**
1. **Workflow JSON is Recoverable**: Even after environment deletion, complete workflow logic preserved in solution export JSON. This enabled full recovery without rebuilding from memory.

2. **Solution Versioning Strategy**: Maintaining multiple solution versions (v1.3.0.1, v1.3.0.2, v1.3.0.3) in `Solution_Exports/` folder enabled comparison and rollback options.

3. **Test Data Configuration is Critical**: Complex calculated field dependencies require careful test data setup. Documentation of field calculation logic essential for debugging.

### **Business Decisions**
1. **Continue with v1.3.0.3**: User chose to debug existing environment rather than import v1.3.0.2 and recreate workflow, preserving "almost finished" work.

2. **Solution Exports in Git**: Decision to track solution .zip files in repository for version control and disaster recovery.
